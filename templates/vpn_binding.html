Binding transmission-daemon to a Mullvad WireGuard Tunnel

Complete setup guide for Ubuntu Server (tested on Raspberry Pi 4, Ubuntu Server 26.04, Transmission 4.x). Following these steps exactly gets transmission-daemon bound to a Mullvad WireGuard interface with split-tunnel routing (SSH/Tailscale untouched), a UID-based kill switch, and boot persistence.

Architecture at a glance:

  • WireGuard interface named mullvad, brought up via wg-quick
  • Table = off prevents the tunnel from hijacking the default route (protects SSH)
  • Policy routing (table 200) makes traffic sourced from the tunnel IPs actually route through the tunnel — required, not optional; binding alone does not work without it
  • Transmission binds to the tunnel IPs (both IPv4 and IPv6)
  • UID-based policy routing forces all of the debian-transmission user's traffic through the tunnel — not only the packets Transmission source-binds — so tracker announces and other unbound connections can't slip out the real link
  • A WireGuard FwMark tags the tunnel's own encrypted packets so the kill switch lets them out to the relay; without it the kill switch drops them and all of the daemon's outbound traffic silently fails
  • iptables/ip6tables kill switch restricts the debian-transmission user to loopback + tunnel only
  • RPC locked to localhost with authentication

Prerequisites

  • Ubuntu Server with root/sudo access
  • A Mullvad account
  • If accessing the machine remotely: keep a second SSH session open during setup as a safety net

Step 1: Install packages

sudo apt update
sudo apt install wireguard wireguard-tools transmission-daemon -y

Verify:

wg --version

Step 2: Generate and download a Mullvad WireGuard config

  1. Go to https://mullvad.net/en/account/wireguard-config and log in
  2. Generate a key / select a device
  3. Pick a server location
  4. Download the .conf file (e.g. at-vie-wg-102.conf)

Step 3: Transfer the config to the server

From your local machine:

scp ~/Downloads/<filename>.conf user@<server>:/tmp/

On the server:

sudo mv /tmp/<filename>.conf /etc/wireguard/mullvad.conf
sudo chmod 600 /etc/wireguard/mullvad.conf
sudo chown root:root /etc/wireguard/mullvad.conf

Step 4: Note your tunnel IPs and the daemon's UID

sudo cat /etc/wireguard/mullvad.conf

Find the Address line. It contains your IPv4 and IPv6 tunnel addresses, e.g.:

Address = 10.75.112.58/32,fc00:bbbb:bbbb:bb01::c:7039/128

These two IPs are used throughout the rest of this guide. Wherever you see <TUNNEL_IPV4> or <TUNNEL_IPV6> below, substitute your actual values (without the /32 or /128 suffix).

Also note the numeric UID of the daemon's user — the policy routing in Step 5 needs it:

id -u debian-transmission

Wherever you see <TRANSMISSION_UID> below, substitute this number (commonly something like 105, but it varies by system).

Step 5: Edit the config — add split-tunnel routing

sudo nano /etc/wireguard/mullvad.conf

Edit it to match this structure, keeping your own PrivateKey, Address, PublicKey, and Endpoint values:

[Interface]
# Device: <your device name>
PrivateKey = <unchanged>
Address = <TUNNEL_IPV4>/32,<TUNNEL_IPV6>/128
Table = off
MTU = 1380
DNS = 10.64.0.1
FwMark = 0xca6c
PostUp = ip route add default dev mullvad src <TUNNEL_IPV4> scope link table 200; ip rule add from <TUNNEL_IPV4> table 200; ip -6 route add default dev mullvad table 200; ip -6 rule add from <TUNNEL_IPV6> table 200; ip rule add fwmark 0xca6c lookup main priority 5255; ip rule add uidrange <TRANSMISSION_UID>-<TRANSMISSION_UID> lookup 200 priority 5260; ip -6 rule add uidrange <TRANSMISSION_UID>-<TRANSMISSION_UID> lookup 200 priority 5260
PostDown = ip rule delete from <TUNNEL_IPV4> table 200; ip route delete default dev mullvad table 200 2>/dev/null || true; ip -6 rule delete from <TUNNEL_IPV6> table 200; ip -6 route delete default dev mullvad table 200 2>/dev/null || true; ip rule del fwmark 0xca6c lookup main priority 5255 2>/dev/null || true; ip rule del uidrange <TRANSMISSION_UID>-<TRANSMISSION_UID> lookup 200 priority 5260 2>/dev/null || true; ip -6 rule del uidrange <TRANSMISSION_UID>-<TRANSMISSION_UID> lookup 200 priority 5260 2>/dev/null || true

[Peer]
PublicKey = <unchanged>
AllowedIPs = 0.0.0.0/0,::0/0
Endpoint = <unchanged>

What each addition does:

  • Table = off — prevents WireGuard from installing itself as the system default route. Without this, all traffic (including your SSH session) gets routed through the tunnel. This is what keeps remote access safe.
  • MTU = 1380 — conservative MTU that avoids fragmentation issues on home-network paths with extra encapsulation layers.
  • PostUp policy routing — because Table = off disables automatic routing, traffic sourced from the tunnel IPs has no route to the tunnel without these rules. This step is mandatory: Transmission's bind-address setting sets the source IP of its packets, but the kernel needs these rules to know that packets from that source IP go out via mullvad. Without them, bound traffic silently goes nowhere.
  • DNS = 10.64.0.1 — Mullvad's internal resolver. Note: if wg-quick up fails with resolvconf: command not found, either install resolvconf (sudo apt install resolvconf) or simply delete this DNS line — it is not required for Transmission to work.
  • FwMark = 0xca6c — tags WireGuard's own encrypted packets with a firewall mark. This is what lets those packets past the kill switch in Step 11: the encrypted ciphertext leaves via the physical NIC to reach the Mullvad relay (not via mullvad), so the kill switch would otherwise drop it. 0xca6c is WireGuard's conventional mark; any unused value works as long as Step 11's ACCEPT rule matches it. Verify after wg-quick up with sudo wg show mullvad fwmark — it must print 0xca6c, not off.
  • fwmark 0xca6c lookup main priority 5255 — routes those marked (encrypted) packets out via the main table (your real NIC) so they reach the relay, instead of being caught by the UID rule below and looped back into the tunnel.
  • uidrange <TRANSMISSION_UID> lookup 200 (IPv4 and IPv6) — routes all of the daemon's traffic through the tunnel, not only what it source-binds. The from <TUNNEL_IPV4> rule above only catches packets Transmission binds to the tunnel IP — it binds its peer sockets there, but not its tracker/HTTP announces (those go out through libcurl unbound). Without this rule, tracker announces route to the real link and the kill switch drops them, so Transmission can never reach a tracker. Priority 5260 keeps it just after the fwmark escape rule (5255) so encrypted packets escape first.

Step 6: Bring up the tunnel and verify

If accessing remotely, open a second SSH session first and keep it open.

sudo wg-quick up mullvad
sudo wg show

wg show may not show a latest handshake line until traffic is sent — WireGuard is lazy and only handshakes on demand. Force it and verify everything at once:

curl --interface <TUNNEL_IPV4> https://am.i.mullvad.net/json

Expected: JSON containing "mullvad_exit_ip": true. If you get this, the tunnel, routing, and DNS are all working.

Verify UDP works through the tunnel too (TCP working does NOT imply UDP works — test both):

dig @8.8.8.8 google.com -b <TUNNEL_IPV4>

Expected: a normal DNS answer, not a timeout. If this times out, the policy routing from Step 5 is missing or wrong — recheck ip rule show and ip route show table 200.

Confirm your SSH session still works by opening one more fresh connection.

Step 7: Enable the tunnel on boot

sudo systemctl enable wg-quick@mullvad.service

Step 8: Configure transmission-daemon

Critical rule for editing settings.json: transmission-daemon rewrites settings.json from memory on every shutdown. If you edit the file while the daemon is running (or not fully stopped), your changes get silently overwritten. Always follow this sequence: 

sudo systemctl stop transmission-daemon
pgrep -af transmission-daemon    # MUST return nothing before you edit
sudo nano /etc/transmission-daemon/settings.json

(Note: use pgrep -af, not pgrep -a — the process name exceeds the kernel's 15-character matching limit, so plain pgrep gives false negatives.)

Set these keys (edit existing ones; add any that are missing):

"bind-address-ipv4": "<TUNNEL_IPV4>",
"bind-address-ipv6": "<TUNNEL_IPV6>",
"peer-port-random-on-start": true,
"port-forwarding-enabled": false,

"rpc-enabled": true,
"rpc-bind-address": "127.0.0.1",
"rpc-authentication-required": true,
"rpc-username": "transmission",
"rpc-password": "YourPasswordHere",
"rpc-whitelist-enabled": true,
"rpc-whitelist": "127.0.0.1",
"rpc-host-whitelist-enabled": true,
"rpc-host-whitelist": "127.0.0.1"

Notes:

  • The bind addresses must exactly match the tunnel's current IPs. If you ever switch Mullvad servers/configs, the tunnel IPs change and these two fields MUST be updated to match, or Transmission cannot connect to anything. This is the single most common failure point.
  • port-forwarding-enabled: false — Mullvad discontinued port forwarding in 2023; Transmission's UPnP attempts through the tunnel are useless. Peer connectivity works via outbound connections, DHT, and PEX.
  • The rpc-password is written in plaintext; Transmission hashes it automatically on next start.
  • Avoid $, ', and " characters in the password — they get mangled by bash when passed via --auth unless carefully quoted.
  • RPC bound to 127.0.0.1 means the web UI/RPC is only reachable from the machine itself (or through an SSH tunnel), never exposed publicly.

Verify your edits saved before starting:

sudo cat /etc/transmission-daemon/settings.json | grep -E "bind-address|rpc-bind"

Step 9: Start and verify bindings

sudo systemctl start transmission-daemon
sudo ss -tlnp | grep transmission

Expected output — three listeners:

  • Peer port on <TUNNEL_IPV4> (NOT 0.0.0.0)
  • Peer port on [<TUNNEL_IPV6>] (NOT [::])
  • RPC port 9091 on 127.0.0.1 (NOT 0.0.0.0)

If the IPv6 listener shows your real LAN IPv6 address instead of the tunnel IPv6, the bind-address-ipv6 field is missing or wrong — that is an IPv6 leak vector; fix it before proceeding.

Step 10: Verify RPC authentication

export TR_AUTH="transmission:YourPasswordHere"
transmission-remote --auth "$TR_AUTH" -l

Expected: an empty torrent list with a Sum: line, no 401 error.

To persist the auth variable across sessions:

echo 'export TR_AUTH="transmission:YourPasswordHere"' >> ~/.bashrc
source ~/.bashrc

(Plaintext in .bashrc — acceptable for a single-user machine, but know the tradeoff.)

Step 11: Add the kill switch (IPv4 AND IPv6)

The bind address alone is not a kill switch — if the tunnel drops and comes back with a different IP, or any code path doesn't respect the binding, traffic could use the real interface. These rules make it impossible: the debian-transmission user can only send packets via loopback or the tunnel.

sudo iptables  -A OUTPUT -m owner --uid-owner debian-transmission -o lo      -j ACCEPT
sudo iptables  -A OUTPUT -m owner --uid-owner debian-transmission -o mullvad -j ACCEPT
sudo iptables  -A OUTPUT -m mark --mark 0xca6c                              -j ACCEPT
sudo iptables  -A OUTPUT -m owner --uid-owner debian-transmission            -j DROP

sudo ip6tables -A OUTPUT -m owner --uid-owner debian-transmission -o lo      -j ACCEPT
sudo ip6tables -A OUTPUT -m owner --uid-owner debian-transmission -o mullvad -j ACCEPT
sudo ip6tables -A OUTPUT -m owner --uid-owner debian-transmission            -j DROP

Rule order matters (evaluated top-down): allow loopback, allow the tunnel, allow the tunnel's own encrypted packets, drop everything else — for that UID only. All other users/services are unaffected.

The --mark 0xca6c ACCEPT rule is essential and easy to miss. WireGuard encrypts the daemon's packets and sends the ciphertext out your physical NIC to reach the Mullvad relay. That ciphertext belongs to the debian-transmission user but does not leave via mullvad, so without this rule the DROP below eats it — and then every outbound connection the daemon makes silently fails (trackers report "Could not connect", and only inbound peers ever work). The FwMark from Step 5 is what tags those packets so this rule can match them. It's IPv4-only on purpose: the relay endpoint is IPv4, so all encrypted packets are IPv4 regardless of whether the inner traffic was v4 or v6.

The IPv6 rules are NOT optional: Transmission binds an IPv6 listener, and without ip6tables rules, IPv6 traffic has no kill switch.

Verify:

sudo iptables -L OUTPUT -v -n
sudo ip6tables -L OUTPUT -v -n

Step 12: Persist the kill switch across reboots

sudo apt install iptables-persistent -y

Answer "yes" to saving both IPv4 and IPv6 rules when prompted. To re-save later after any rule changes:

sudo netfilter-persistent save

Step 13: End-to-end test

Use the official Ubuntu ISO torrent — well-seeded, legal, uses HTTPS trackers:

wget https://releases.ubuntu.com/24.04/ubuntu-24.04.4-live-server-amd64.iso.torrent -O /tmp/ubuntu.torrent
transmission-remote --auth "$TR_AUTH" -a /tmp/ubuntu.torrent
sleep 30
transmission-remote --auth "$TR_AUTH" -t 1 -i

Expected within a minute: State: Downloading, non-zero download speed, Peers: connected to <N> with N > 0.

Confirm the traffic is genuinely using the tunnel:

sudo ss -tn state established | grep <TUNNEL_IPV4>

Expected: established connections with <TUNNEL_IPV4> as the local address. (Note: ss -tnp with process matching may show nothing due to the daemon's systemd sandboxing — filter by IP instead, as shown.)

Confirm the kill switch counters are live:

sudo iptables -L OUTPUT -v -n

The mullvad ACCEPT rule's packet counter should be climbing during the download.

Step 14: The definitive kill switch test

Prove traffic fails closed when the tunnel drops:

sudo wg-quick down mullvad
sleep 5
sudo ss -tn state established | grep <TUNNEL_IPV4>    # should show nothing
sudo wg-quick up mullvad                               # restore

With the tunnel down, Transmission must show zero peer connections and zero transfer — never activity on your real IP.

Ongoing maintenance

If you ever change Mullvad servers

The tunnel IPs change with every new config. You MUST update, in this order:

  1. /etc/wireguard/mullvad.conf — new Address, PublicKey, Endpoint, and the IPs inside PostUp/PostDown
  2. /etc/transmission-daemon/settings.json — bind-address-ipv4 and bind-address-ipv6 (stop the daemon fully first, per Step 8)
  3. Restart both: sudo wg-quick down mullvad && sudo wg-quick up mullvad, then sudo systemctl restart transmission-daemon

The kill switch rules reference the interface name (mullvad), not IPs, so they never need updating.

Quick health check (run any time)

sudo wg show                                          # recent handshake?
sudo wg show mullvad fwmark                            # 0xca6c (not "off")?
curl --interface <TUNNEL_IPV4> https://am.i.mullvad.net/json   # mullvad_exit_ip: true?
sudo ss -tlnp | grep transmission                     # bound to tunnel IPs?
ip rule show && ip route show table 200               # policy routing intact?

Troubleshooting: every tracker shows "Could not connect"

If DHT finds peers but every tracker announce fails with "Could not connect", the daemon's outbound packets are being dropped by its own kill switch — specifically WireGuard's encrypted packets, which must leave via the real NIC. Check that all three pieces from Steps 5 and 11 are present:

sudo wg show mullvad fwmark                              # expect: 0xca6c (not "off")
ip rule show | grep -E 'uidrange|fwmark 0xca6c'         # expect the 5255 + 5260 rules
sudo iptables -C OUTPUT -m mark --mark 0xca6c -j ACCEPT && echo present   # the ACCEPT must exist

Then confirm the daemon's own user can reach the internet through the tunnel:

sudo -u debian-transmission curl -sS --max-time 15 https://am.i.mullvad.net/ip

This must print your Mullvad exit IP. If it times out, one of the three pieces is missing. To see exactly where a packet dies, run sudo tcpdump -nni any host <relay-ip> (relay from sudo wg show mullvad endpoints) while repeating the curl — if the encrypted UDP never leaves for the relay, the fwmark/ACCEPT chain is the problem; if it leaves but nothing returns, look at the routing/bind.

Known limitations

  • No inbound port forwarding — Mullvad discontinued it. Inbound peers can't reach you; peering relies on outbound-initiated connections, DHT, and PEX.
  • Tracker errors on individual trackers are normal — public torrents often list many trackers, several of which are dead or unreliable at any given time. As long as the torrent finds peers (via any working tracker, DHT, or PEX), a few individual tracker errors are cosmetic.

But if every tracker fails with "Could not connect" — not just a few — that is not a limitation, it's the fwmark/UID-routing setup from Steps 5 and 11 missing or wrong. See the troubleshooting entry above. (Earlier versions of this guide omitted the fwmark step and wrongly treated blanket tracker failure as normal; it isn't — the daemon's outbound was being dropped by its own kill switch.)
