Media server (copy over SSH)

Where the per-torrent Copy to media server action sends files. Uses rsync over SSH — set up key-based auth from this host to the media server (e.g. ssh-copy-id user@host) so it can run without prompting.

Can't connect? Setting up SSH access to a new server

An error like user@100.64.1.5: Permission denied (publickey,password) means the network is fine — SSH reached the server — but it rejected the login. That's a credentials problem, not connectivity. Work through these:

  • Check the username. The User field must be a real account on the media server (e.g. the OS default or the one you created during install), not a placeholder like user.
  • Install this host's key on the server. Publickey fails until the new server has this host's public key. From this host, run once: ssh-copy-id user@100.64.1.5 — it prompts for the password, then appends the key to ~/.ssh/authorized_keys. Key auth works after that.
  • If the password is also rejected, either it's the wrong user/password, or the server has PasswordAuthentication no in its sshd config. In that case ssh-copy-id can't bootstrap — add this host's key another way (console/keyboard access, or the server's own admin UI).
  • Tailscale/CGNAT note: addresses in 100.64.0.0/10 are Tailscale IPs. Make sure the new server is actually on your tailnet and reachable (tailscale ping) before assuming an auth problem.

How to choose a free space margin

The margin is a percent of each disk's total size that copies must leave free (8% of a 1 TB disk keeps ~80 GB in reserve). It absorbs source-size estimation error and stops a copy from filling a disk outright. Rules of thumb:

  • Disk that also runs the OS or media server app (e.g. the server's root filesystem): keep 8–10%. The app's database, metadata and logs live there — if it fills up, the media server itself starts failing, not just copies.
  • Dedicated media-only disk: 5% is plenty. Media files are written once and read many times, so SSD wear and near-full write slowdown matter little.
  • 0% disables the buffer — a copy may fill the disk to the last byte.

In a fallback group each drive is judged against its own margin, so a stricter margin on one drive simply sends copies to the next one sooner.

Give each folder a label (shown in the dropdown when copying) and the absolute path on the media server. Tracker IP leak test

End-to-end check of what a real BitTorrent tracker sees: adds a test magnet from an IP-echo service, reads the IP the tracker reports back, and compares it against the tunnel's exit IP. Complements the tunnel indicator — this is the only check that exercises Transmission's actual announce path. When enabled, a Run tracker IP test button appears on the System page's Tunnel card.

Off by default: running the test adds a (data-less) test torrent to Transmission and contacts two external services.

Paste a magnet from an IP-echo service — one whose tracker reports the IP it saw inside the tracker status message. TorGuard's Check My Torrent IP page generates one; ipleak.net's Torrent address detection magnet also works (its per-session results additionally appear on their page). The magnet must include a tracker (tr=). A regular torrent's magnet (an Ubuntu ISO, say) will not work — ordinary trackers don't report the IP they saw, so there'd be nothing to check. Nothing is ever downloaded: the test torrent is stopped the moment the tracker answers and removed when the test ends.

Periodic runs happen server-side (no browser needed) and log their outcome to the event history — a detected leak is recorded as an error event. Manual runs reset the timer.

A plain-text what's-my-IP endpoint (default am.i.mullvad.net/ip; api.ipify.org also works). The dashboard calls it with the connection source-bound to the tunnel address to learn which exit IP the tracker should have seen.
