Running the Dashboard as a systemd Service

Setup guide for running the dashboard under gunicorn + systemd so it starts on boot, restarts on failure, and logs to the journal. A ready-made unit template ships with the repo at systemd/transmission-dashboard.service.example.

Prerequisites

  • The repo checked out on the server (e.g. /home/<user>/transmission-dashboard)
  • A configured .env in the repo root (copy .env.example and fill it in)
  • Root/sudo access

Step 1: Create the virtualenv

The unit expects a virtualenv at <repo>/.venv with the dependencies (including gunicorn) installed:

cd /home/<user>/transmission-dashboard
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt

Step 2: Install the unit file

sudo cp systemd/transmission-dashboard.service.example \
  /etc/systemd/system/transmission-dashboard.service
sudoedit /etc/systemd/system/transmission-dashboard.service

Replace every CHANGE_ME:

  • User= / Group= — the account that owns the checkout. Do not run it as root; the dashboard needs no elevated privileges beyond what the unit already grants (see Step 4).
  • WorkingDirectory= — the absolute path of the repo.
  • EnvironmentFile= — the absolute path of the .env inside the repo.
  • ExecStart= — the same repo path in the gunicorn binary's path.

Step 3: Leave the worker settings alone

The unit runs gunicorn with one worker and eight threads (--workers 1 --threads 8). This is intentional, not a placeholder to tune up:

  • This is a single-user dashboard — extra workers add no capacity.
  • Scan/copy state lives in-process. With multiple workers, a /api/scan/<id>/stop request can land in a different worker than the one running the scan and return 404, because the cancel event lives in the other worker's memory. Custom names, copy state, and caches fragment the same way.
  • There is deliberately no --access-logfile: the UI polls several endpoints per second, and logging every poll wrote a journald line each — constant CPU/I/O for zero diagnostic value. Errors still reach the journal via --error-logfile -.

Step 4: The WireGuard capability pre-step

The unit contains this block:

ExecStartPre=+/bin/sh -c '/sbin/setcap cap_net_admin+ep "$(command -v wg)" || true'
AmbientCapabilities=CAP_NET_ADMIN
CapabilityBoundingSet=CAP_NET_ADMIN

What it does and why it exists:

  • The tunnel indicator runs wg show, which makes a netlink call requiring CAP_NET_ADMIN. Without the capability, the check sees no handshake and the indicator always reports "Tunnel down" even when the tunnel is fine.
  • The setcap re-applies on every start because an apt upgrade of wireguard-tools replaces the binary and strips its file capabilities — without the pre-step this silently regresses.
  • The leading + runs the pre-step as root regardless of User=; || true keeps the unit bootable if wg is missing.
  • Only relevant if you use the tunnel indicator (TUNNEL_IFACE set in .env) — harmless otherwise; leave it in place either way.

Step 5: Enable and start

sudo systemctl daemon-reload
sudo systemctl enable --now transmission-dashboard

Step 6: Verify

systemctl status transmission-dashboard
sudo ss -tlnp | grep 5000

Expected: active (running), and gunicorn listening on 127.0.0.1:5000 (NOT 0.0.0.0 — the dashboard is meant to be reached via localhost, an SSH tunnel, or a reverse proxy you control).

Then open the dashboard in a browser and confirm you get the login page.

Ongoing maintenance

Logs

journalctl -u transmission-dashboard -f          # follow live
journalctl -u transmission-dashboard --since -1h # recent history

Updating the dashboard

Use the repo's update script — it pulls, reinstalls dependencies into .venv, and restarts the service:

cd /home/<user>/transmission-dashboard
./update.sh

Or manually:

git pull --ff-only
.venv/bin/pip install -r requirements.txt
sudo systemctl restart transmission-dashboard

After editing .env

systemd reads EnvironmentFile= only at service start — a restart is required for changes to take effect:

sudo systemctl restart transmission-dashboard

If the service won't start

  • systemctl status transmission-dashboard shows the last few log lines and the exit reason; journalctl -u transmission-dashboard -e shows the full traceback.
  • Missing .env — a nonexistent EnvironmentFile= path fails the unit before gunicorn even runs. Prefix the path with - (EnvironmentFile=-/…/.env) only if you intend it to be optional.
  • status=203/EXEC — the gunicorn path in ExecStart= is wrong or the virtualenv is missing; re-check Step 1 and the paths from Step 2.
  • Restart loop — the unit uses Restart=on-failure with a 5-second delay, so a crashing app restarts forever; fix the underlying error from the journal rather than the unit.
